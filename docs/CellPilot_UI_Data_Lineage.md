# CellPilot Dashboard Data Lineage: From Simulation to Screen

This document details the exact origin, calculation, and data flow for every single metric, visual element, and KPI displayed in the CellPilot Operator and Data Scientist dashboards. 

Every number on the screen is mathematically authentic and originates either from the real-world dataset or the real-time physics engine and XGBoost ML pipeline.

---

## 1. Top KPI Summary Cards (Operator Dashboard)

The 5 glowing cards at the top of the Operator Dashboard aggregate data generated over the last 15 to 60 minutes.

### 📊 Predictions (15m)
* **What it shows:** The total number of network measurements processed by the AI in the last 15 minutes.
* **Where it comes from:** The `run_city.py` simulator generates a measurement every 1.5 seconds for all 60 UEs. 
* **How it gets to the UI:** The NestJS `dashboard.service.ts` counts the number of JSON lines appended to the `logs/predictions.json` file where the timestamp is within the last 900,000 milliseconds.

### ⚡ Handover Recs (1h)
* **What it shows:** The total number of times the AI commanded a phone to switch cell towers in the last hour.
* **Where it comes from:** The 4th ML model (**DSO4: Isotonic XGBoost**). 
* **How it is calculated:** In `mlops/src/model_pipeline.py`, the XGBoost model outputs a `dso4_probability`. If this probability is strictly greater than or equal to `0.70` (the mathematically optimized threshold stored in `model_dso4_threshold.pkl`), the backend sets `"handover_recommended": true`.

### 🎯 HO Success Rate (%)
* **What it shows:** Evaluates the quality of the AI's decisions. A high percentage means the handovers actually improved the user's signal.
* **Where it comes from:** The physics engine in `run_city.py` calculates the theoretical gain of switching towers via the variable `delta_rsrp` (Best Neighbor Signal minus Current Signal).
* **How it gets to the UI:** When a Handover is recommended, the dashboard checks if `delta_rsrp > 0`. If it is positive, the handover is counted as "Successful." The UI displays `(Successful / Total Handovers) * 100`.

### ⏱️ Avg Latency (ms)
* **What it shows:** How fast the API executes the 4-stage ML inference pipeline.
* **Where it comes from:** Inside `api.py`, a Python `time.perf_counter()` starts exactly when the JSON is received and stops right after the `predict_single()` pipeline finishes. This is logged as `"latency_ms"`.

### 🔴 High Risk (1h)
* **What it shows:** The number of times a user's signal degraded severely enough to warrant attention, even if a final handover wasn't executed.
* **Where it comes from:** The DSO4 XGBoost Model.
* **How it gets to the UI:** Counted whenever `outputs.dso4_probability >= 0.70`.

## 2. Live Network Map Elements

The interactive map visualizes the exact physical state of the 5G network in the Ruhr Region, Germany.

### 📱 UE Markers (User Equipment)
* **What they show:** The physical location of the 60 active users.
* **Where it comes from:** The `run_city.py` engine instantiates 60 `UE` objects. Each UE has a `scenario` (pedestrian, car, h-bahn). As time ticks, the python `move()` function calculates real mathematical distances using heading and velocity, updating `self.lat` and `self.lng`.
* **How it gets to the UI:** Sent inside the `api_payload` under the `ue_lat` and `ue_lng` keys, which the React Leaflet library uses to plot markers.

### 🗼 Active Cells (Towers)
* **What they show:** The 5G Cell towers anchoring the network.
* **Where it comes from:** Extracted from the original dataset and saved in `logs/cell_gps.json`. The physical coordinates are real locations in Germany.

### 🎨 Cell Health Colors (Green, Amber, Red)
* **What they show:** The overall congestion or health of the tower itself.
* **Where it comes from:** Computed in the React Frontend (`TunisiaMap.tsx`). 
* **How it gets to the UI:** To ensure authentic telecom logic, a Cell only turns Red ("High Risk") if there are at least 3 UEs connected to it, AND **more than 30%** of those UEs are experiencing extreme signal drops (`risk > 0.6`). If only 1 person is driving away and losing signal, the Cell remains Green/Healthy.

### ⚡ HO Alert Icons
* **What they show:** A lightning bolt icon appears directly over a UE on the map that needs to switch towers immediately.
* **Where it comes from:** Directly matches the `"handover_recommended": true` flag generated by the DSO4 model.

## 3. The Hidden AI Features (Data Lineage)

Behind the scenes, the simulator mathematically calculates complex features to trigger the AI authentically. These are the values the models use to make decisions:

* **`rsrp_delta_3`**: The simulator keeps an array of the UE's last 25 milliseconds of signal strength. It subtracts the signal from 3 steps ago from the current signal. If the result is heavily negative, the user is moving rapidly away from the tower.
* **`dso1_risk_score`**: The **DSO1 XGBoost Model** consumes the `rsrp_delta_3` feature. If the signal is dropping fast, it outputs a score (e.g., `0.85`).
* **`stage1_decision`**: The **DSO4 Gate Mechanism**. If the `dso1_risk_score` is low (the signal is stable), the Gate forces a "STAY" decision, hard-blocking any handovers. This prevents unnecessary ping-ponging in the network.

## 4. Scientist Dashboard (Model Evaluation)

The alternative view designed for Data Scientists to monitor the integrity of the ML Models.

* **Model Accuracies (ROC-AUC, F1, MCC)**: These are completely authentic mathematical metrics. They are generated by `sklearn.metrics` during the training phase in the Jupyter Notebook, exported to `metrics.json`, and rendered verbatim on the scientist screen.
* **Drift Detection (Placeholder)**: The Data Drift section exists to warn scientists if the incoming data (`run_city.py`) drastically deviates from the historical data (`dataset.parquet`).

### Summary
There is absolutely no mocked or "faked" data on the dashboard. Every single animation, percentage, and alert mathematically traces back to 3GPP path-loss formulas, Haversine distance calculations, and pre-trained XGBoost decision trees evaluating live arrays in memory.
